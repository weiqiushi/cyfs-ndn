# Content Network

> 本文讨论的不是“如何传一份字节流”，而是“一个内容如何被命名、发布、发现、传播、消费、评价与再创作”。

当前文档状态：0.9，最新草案，与 `CYFS Protocol.md` 配套阅读。

## 设计目标与边界

互联网最重要的功能之一，就是发布内容。

如果只把互联网理解成“端到端传输字节流”的网络，那么我们只能解释 HTTP、TCP、TLS 这些基础设施；但真实世界里更重要的问题是：

- 谁可以发布内容。
- 用户如何发现值得看的内容。
- 内容在传播过程中，谁贡献了发现、整理、背书和继续扩散的价值。
- 当内容真的被消费、安装、使用甚至产生收入时，这些价值如何被公平地确认。

`Content Network` 讨论的正是这一层语义。

它的根本目标有四个：

- 保障发布内容，以及通过内容直接获得收入的权利。
- 建立公平而透明的内容传播规则，让规则里没有平台特权。
- 让这套规则真正鼓励人们持续创造更多、更高水平的内容。
- 帮助用户在噪音极大的网络中，更高效地看到值得信任的内容和观点。

这里有一个很重要的边界：

> `Content Network` 不是中心化平台的“内容运营手册”，而是一个去中心化内容网络应当具备的语义层模型。

它主要回答下面这些问题：

- 一个内容在网络中的“利益相关方”是谁。
- 这些角色各自通过什么对象与行为留下可验证事实。
- 上层 App / Agent 如何基于这些事实构造发现、分发、评级和收益分配。

它**不**试图在协议层一次性解决所有问题，例如：

- 不定义唯一的全局推荐算法。
- 不定义唯一的内容审核机构。
- 不承诺存在一个全网唯一正确的“内容质量分数”。
- 不把所有传播都简化成“点击量”和“转发数”。

一句话：**Content Network 关心的是内容传播的语义与价值流，不是中心化平台的控制流。**

## 核心角色

任何一个 `FileObject` 或更一般的 `NamedObject`，在网络中的发布与传播，至少涉及四类核心角色。

### Author 与 Owner

第一类角色是内容的创造者与权利主体：

- `author: String`，作者，可以为空；为空时，作者默认等于 `owner`。
- `owner: DID`，所有者；为空时表示 `copyleft`，即不主张排他的所有权。

几种常见情况：

- `author` 和 `owner` 都存在：表示“谁创作”和“谁持有权益”被明确区分。
- 只有 `owner`：表示该对象的利益主体明确，但未单独声明作者。
- 两者都为空：表示一个匿名且无版权约束的对象。

这里的 `Owner` 是整个收益分配与权利关系的原点。  
但 `Content Network` 有意把“权利主体”和“流量入口”区分开来：作者并不天然拥有唯一的传播入口，传播价值还会继续分配给后续角色。

### Curator / 收录者

第二类角色是 `Curator`，也就是收录者。

收录者的核心语义是：

> 某个主体明确地把某内容纳入了自己的目录、集合、榜单、应用商店、专题推荐或其它发现上下文。

收录者未必保存在对象内部。更常见的情况是，它存在于某个外部 `context` 里：

- `DirObject` 是一种标准的收录 context。
- `pkg-source` 可以是一种基于 SQLite 数据库的收录 context。
- 搜索引擎、专题站、榜单站、AI curated list，本质上也都可以被视为某种 curator context。

从关系方向上看，收录可以分为两类：

- **单向收录**：收录者声明“我收录了这个内容”，不需要作者预先认可。
- **双向收录**：除了收录者声明之外，`Owner` 也对这段收录关系签名确认。

这两者在“是否允许表达”上都成立，但在收益分成和品牌背书上会有区别。  
例如，一些强品牌 curator 可能只愿意传播“双向收录”的对象。

### Sharer / 传播者

第三类角色是 `Sharer`，也可以叫传播者。

传播者通常**不保存在内容对象内部**。  
它通常是基于某个传播 context 推导出来的，例如：

- Alice 把一个对象分享给 Bob，那么 Alice 就是这条路径上的传播者。
- 某个群聊管理员把内容转进群里，那么该管理员或该群对象可以成为传播路径的一部分。
- 某个 Agent 把内容推荐给另一个 Agent，那么这个 Agent 也可以构成传播者。

在一些场景下，传播者也可以主动构造签名证明，声明“我传播过这个内容”；但很多时候，这类证明只需要设备私钥或会话私钥即可，不需要把传播者写进内容对象本身。

### Consumer / 消费者

第四类角色是 `Consumer`，也就是最终真正获取、安装、停留、使用并可能继续传播内容的人或 Agent。

这是内容网络里最关键的一类反馈来源，因为：

- `download` 说明至少发生过获取。
- `installed` 说明发生了更高成本的接纳。
- `shared` 说明发生了主动再传播。
- 更深层的 `content consumption proof` 则说明出现了真实的时间与注意力投入。

因此，消费者不是一个被动终点，而是下一轮传播和收益分配的起点。

### 小结：四类角色的关系

可以把这四类角色理解为内容网络中的四个价值层：

| 角色 | 语义 | 价值来源 |
| --- | --- | --- |
| `Author / Owner` | 内容供给的原点 | 创作与权利 |
| `Curator` | 内容发现与筛选 | 收录、排序、背书 |
| `Sharer` | 传播路径扩散 | 触达新受众 |
| `Consumer` | 真实需求反馈 | 获取、安装、使用、继续传播 |

`Content Network` 的设计意图，不是让其中某一个角色垄断全部利益，而是让这四类角色都能留下可计算、可验证的事实。

## 发布：名字是内容的入口

当打算严肃地发布一个内容时，首要工作通常不是“上传文件”，而是“给内容起名字”。

这是因为在内容网络里，真正长期有价值的不是“某次下载地址”，而是：

- 用户如何记住这个内容。
- 用户如何再次找到它。
- 其它 curator、sharer、search engine、AI agent 如何引用它。

每个内容创作者都可以基于去中心化身份基础设施，拥有自己的 `DID`，例如：

```text
did:bns:alice
```

在此基础上，可以给某个内容或某个内容空间分配更有意义的名字，例如：

```text
did:bns:app1.alice
did:bns:alice:app1
```

其核心要求不是“全球唯一字符串形式必须固定”，而是：

> 在某个 `Owner` 已发布的命名空间里，这个名字应当能稳定地指向一个明确的内容入口。

这里要区分两层：

- **有意义的名字**：通过 `did resolve` 或等价系统解析到 `did document` 及相关发布信息。
- **不可变的内容对象**：通过 `NamedObject` / `ObjectId` 体系精确定位到某个确定版本的内容。

所以一次标准发布，逻辑上通常包括两步：

1. 让用户或 agent 可以通过一个“有意义的名字”找到发布入口。
2. 让发布入口在某一时刻指向一个确定的不可变对象。

这与 `CYFS Protocol.md` 中“语义 URL + ObjectId”的分层是完全一致的。

### 发布到 Owner 的源站

内容发布后，原则上至少应当能从 `Owner` 自己控制的源站或 `OOD` 获取。  
例如，某个对象发布到 Alice 的 OOD 后，可以通过其 Zone 提供的入口完成下载。

这一步非常像传统互联网里的“搭建个人网站”，但又多了一层能力：

- 有意义的名字用于发现。
- `NamedObject` 用于可信定位。
- 后续传播链可以在不改动原始内容对象的前提下不断扩展。

换句话说，**原始发布点解决“内容从哪里开始存在”，而内容网络解决“它如何继续流动”。**

## 内容发现与传播路径

知道名字的人，理论上总可以从原始发布者处直接获取内容。  
但真实世界里，大多数内容并不是这样被发现的。用户更常通过下面几类路径接触到内容：

- **目录网站 / App Store 模式**：通过专业 curator 构造的内容目录来发现内容。
- **搜索模式**：搜索引擎本质上也是一种 curator，只是其收录、排序和呈现方式不同。
- **社交网络分享**：朋友、社区、群组、媒体帐号把内容带入新的传播路径。

在 AI 时代，还会出现新的内容获得方式：

- **Spider 模式**：AI 基于社交网络、浏览记录和用户偏好，主动到原创者处抓取用户可能感兴趣的内容。这是 AI 驱动的拉模式。
- **Agent 交换模式**：Agent 与 Agent 之间基于经济模型直接交换内容、摘要、引用和结果。这一模式里既有 Pull，也可能存在更主动的 Push。

### 传播不是协议广播，而是逻辑传播

`Content Network` 有一个很重要的建模原则：

> 我们更关心“逻辑传播”而不是“协议层广播”。

所谓逻辑传播，意思是：

- 协议里出现的每一次“内容扩散”，最好都能对应某个真实用户动作或真实 Agent 动作。
- 传播价值来自“谁发现了内容、谁把它带进了新的上下文、谁为它做了信用背书”，而不是单纯来自底层网络里发了多少个包。

这也是为什么 `Content Network` 里，`Curator` 与 `Sharer` 的角色如此重要。  
它们不是附属信息，而是发现网络和传播网络的核心节点。

## 经济模型：围绕消费成功分配利益

`Content Network` 的经济模型，本质上也是围绕“名字”和“消费成功”展开的。

最重要的利益原点是：

> 用户成功消费内容。

这里的“消费”可以是非常轻的，也可以是非常重的：

- 下载一个对象。
- 安装一个应用。
- 阅读一篇文章并停留足够长时间。
- 购买一个受保护内容。
- 在使用后继续分享、评论、引用或再创作。

`buckyos` 希望通过简单透明的规则，把由“成功消费”带来的收益，分配给：

- 内容作者 / 权利主体。
- 收录者。
- 分享者 / 传播者。
- 在可计算内容场景下，还可能包括提供算子或算力的节点。

更深一层的利益起点，是：

> 用户实际在内容上花费的时间与注意力。

这需要更强的“内容消费证明”，而不只是一个简单下载日志。

### 经济模型依赖的基础能力

为了让这套机制可运行，至少还需要下面几类基础能力：

- 所有权转让。
- 部分拥有所有权，成为永久股东或临时股东。
- 确认经济收入并据此分红。
- 确认真实用户使用，并获得生态激励。

这里的“各种证明”，本质上是在实现一种 `use to earn` 机制：  
只有真实使用、真实传播、真实带来价值的行为，才应该获得稳定收益。

### 经济模型的设计原则

在协议层，我们更强调下面三条原则：

- **透明**：收益分配应尽量建立在公开可验证的传播事实之上，而不是平台黑箱。
- **角色解耦**：作者、收录者、传播者的价值应分开表达，而不是全部吞并到平台角色里。
- **真实消费优先**：下载量可以是信号，但不能直接等价为最终收益依据。

## 内容评级

去中心化内容网络没有中心化审核平台，也就不应假设存在一个“官方评分”。  
那么用户如何在消费前判断内容质量？

理想情况下，应当至少有三层信息：

- 创作者自己给出的标签与描述。
- 多个 curator 或用户给出的结构化评级。
- AI 基于这些评级时间线和用户偏好，整理出来的动态视图。

因此，用户看到的最终评级，不应是某个平台的唯一分数，而应是：

> 基于多个角色、多个来源、多个时间点的评级事实，经由用户自己的算法整合后的结果。

不同用户的加权规则可以不同，这本身就是内容网络多样性的一部分。

### 主动评级

主动评级是最直接的一类。

- 对收录者来说，为收录内容构造评级，本来就是其核心价值之一。
- 内容收录者可以构建 App，收集其它用户的评级并继续整理。
- 用户自己发表的评级，也是一种轻量内容，同样遵循“发布、收录、分享”的路径。

这意味着“评级”不是平台私有数据库里的一个字段，而是可以被发布、引用、聚合和再传播的内容对象。

### 基于行为的被动评级

除了主观打分，行为本身也会产生评级信号，例如：

- 是否获得过内容。
- 获得内容后的使用情况。
- 是否愿意继续分享或推荐。

但这类信号天然涉及隐私，因此通常更适合用 `ZK` 或其它隐私保护方式传播，而不是把行为日志明文公开。

### 评级系统的反机器人设计

只要评级系统有经济价值，就一定会出现机器人和刷量问题。  
因此，评级系统最终必须能回答一个更深的问题：

> 谁的评级来源本身值得相信？

这其实已经接近现实世界的信用系统。  
建立 `Owner / User / Curator` 的信用评级体系，是比“给内容打分”更难、也更基础的基础设施。

这套系统的设计原则应该至少包括：

- 通过多个阶段的行为进行交叉验证。
- 引入高信用种子节点，但不让它们形成永久特权。
- 如果一个主体不公开足够多的行为事实，就很难获得更高信用。

换句话说，养号的成本应当足够高：  
必须把自己的行为嵌入广泛的网络，并被其它节点长期认可，而不是靠一次性脚本刷出来。

## 二次创作与评论

基于一个已有内容创作新内容，是 `Content Network` 里非常重要的一类活动。  
最常见的二创是：

- 评论。
- 弹幕。
- 摘录。
- 带解释的转发。
- 基于原内容继续创作的新作品。

这里有一个基本原则：

> 我们既保障每个人公开发布内容的权利，也保障每个人对已发布内容进行评价和再创作的权利。

这也是对传统中心化平台“控评”逻辑的反思。  
“控评”本质上是一种平台对传播路径的柔性垄断，也是内容生态不公平的重要来源。

### 发布二创内容的方法

二创内容与常规内容发布流程基本相同，但通常有两个区别：

- 它往往不需要绑定一个长期稳定的全局名字。
- 它往往是轻量级的、变化频繁的，且未必直接追求收入。

因此，很多二创内容更适合保存在特定路径或特定 context 下，例如：

```text
did:bns:$content_name.$owner_name:comments:$comment_objid
```

当用户完成一条评论后，理论上可以把评论投递给多方：

- 自己的 OOD。
- 内容的 `Owner`。
- 内容的收录者（可能有多个）。
- 内容的传播路径上的上游节点。
- 如果该内容已经有评论列表，还可以尝试投递给已参与讨论的用户集合。

这里不要求所有投递都成功；它表达的是一种“评论如何沿着已有内容网络继续扩散”的机制。

### 通过名字或列表聚集二创内容

按上面的路径，内容 `Owner` 和各类 `Curator` 都可以逐渐收集到大量二创内容，并通过常规列表展示这些内容。  
他们当然可以对列表做过滤，但过滤之后仍然只是一种“视图”，而不是对其它评论存在性的消灭。

在 AI 和 OOD 的帮助下，用户还可以对近期参与过讨论的内容启动 Spider 模式，主动读取更多参与者持有的评论列表，并在本地 AI 帮助下整理出一个更完整的视图。

所以对二创内容而言，关键不在于“是否存在一个中心评论区”，而在于：

- 评论本身是可发布对象。
- 评论列表可以有多个。
- 用户可以跨多个列表聚合自己的阅读视图。

## 名字意义的演化与共识性修改

历史上，名字的意义并不是永远不变的。

同一个名字在不同时间、不同社区、不同 curator 语境里，可能会逐渐指向不同的内容集合、不同的解释框架，甚至不同的价值判断。

因此，`Content Network` 不应假设：

- 名字的语义一旦定义就永远冻结。
- 某个中心机构有权永久宣布“这个名字现在只能这样解释”。

更合理的做法是把“名字的意义变化”也理解为内容网络中的一个公开过程：

- 原始 `Owner` 可以继续更新自己对名字的绑定。
- 不同 curator 可以围绕同一个名字形成不同的解释与目录。
- 用户与 Agent 再根据自己的信用模型，决定信任哪一条解释链。

这意味着：

> 名字的共识不是靠覆盖旧内容得到的，而是靠不断追加新的绑定、新的收录、新的解释和新的使用事实逐步形成的。

## 引用内容进行创作

除了评论和再传播，还有一种很重要的创作方式：引用已有内容来支持新的内容。

最典型的例子是论文引用，但这套思想并不限于学术场景。  
在 AI 时代，摘要、蒸馏、摘录、拼接、数据处理、模型输出，都会广泛依赖“我基于哪些已有内容产生了这个结果”。

这里的“引用”并不等同于二创。  
它更强调：

- 新内容与旧内容之间存在明确可追踪的依赖关系。
- 依赖关系本身也是有价值的信息。

### 基于算子的引用

已知两个数据 URL：

```text
cyfs://objurlA
cyfs://objurlB
```

再已知一个算子对象：

```text
calc_obj_1
```

那么可以定义一个结果 URL：

```text
result_url = calc_obj_1(cyfs://objurlA, cyfs://objurlB)
```

此时，用户理论上可以直接：

```text
open_reader(result_url)
```

这就是 `Content Network` 的 `lazy eval` 思路：  
内容不一定是一个已经持久化好的静态文件，它也可以是某个确定性计算过程的结果。

### 为什么引用关系很重要

在传统互联网里，人们更关注“发布了什么数据”。  
但在 `Content Network` 里，“数据之间的关系”同样重要，原因包括：

- 它让内容来源和依赖路径更可追踪。
- 它让中间结果不必总是被提前物化。
- 它让分布式调度器可以基于关系图，更快地决定数据与计算的放置策略。

换句话说：

> 未来操作系统不仅要能 `OpenFile`，还要能 `OpenURL`，并且把“打开一个结果”理解成“按需展开一条内容与计算关系链”。

### Content Network 所期望的 API

从应用接口的角度，`Content Network` 更期待下面这种统一抽象：

- 所有需要数据的地方，都尽量使用 `URL`，而不是只能使用本地文件路径。
- 对数据的消费应当允许 `lazy eval`。
- 发布数据重要，发布数据之间的关系同样重要。
- 系统可以通过跟踪这些关系，大幅加快分布式处理与缓存复用。

## 传播证明

`Content Network` 的传播证明，并不是一个单一格式的“万能凭证”，而是一组可以组合起来的可验证声明对象。

它要证明的不是“世界真相”，而是低成本、可计算地证明：

- 谁在什么时候，对哪个内容，做过什么高价值动作。
- 哪个可信主体收录、推荐、引用、再创作或继续传播了这个内容。
- 一条传播链上，哪些动作只是低成本转发，哪些动作代表了更高成本的信用背书。

这正对应 AI-Native 场景下的核心诉求：  
传播不再只是“转发次数”，而是一个结构化的注意力分配过程。

### 承载传播语义的标准对象

CYFS 当前设计了一些标准对象来承载这套语义：

- `cyact` / `ActionObject`：表达“某主体对象对某目标对象做了某个动作”。
- `cyinc` / `InclusionProof`：表达“某个 curator 把某内容收录进自己的目录或集合”。
- `cyrel` / `RelationObject`：表达“两个内容对象之间存在某种弱关系”，例如 `same`、`part_of`。
- `cymsg` / `MsgObject`：表达“某次传播是通过什么消息上下文发生的”。
- `cymsgr` / `MsgReceiptObj`：表达“这条消息被谁接收、接受、隔离或拒绝”。

这些对象的共同特点是：

1. 它们首先是 `NamedObject`，因此本身可哈希、可缓存、可引用。
2. 它们描述的是传播相关事实，而不是直接承载传播算法。
3. 真正的信任来自“对象内容 + 签发者身份 + 分发上下文”的组合，而不是单看某个字段。

从语义上说，CYFS 想表达的不是“平台给内容打分”，而是把传播拆成一组可计算的原子事实，例如：

- `download`
- `installed`
- `shared`
- `purchased`

然后由上层 App / Agent 根据主体信用、传播路径、内容类型、时间窗口和支付关系，去自行计算权重。

### 下载证明

下载证明的核心语义是：

> 某个主体至少完成过一次对指定内容的获取。

它比浏览更强，但比安装更弱。  
它通常由 `cyact` 承载。

在内容网络和多源加速网络里，下载证明之所以重要，是因为它既能作为“这个内容确实有人要”的需求信号，也能作为“这个节点最近接触过这份内容”的供给线索。

这里要强调一个设计意图：

> CYFS 并不要求每个下载都由中心服务器记账。

下载证明完全可以由下载方、网关、缓存节点、代理 Agent 分别签发为独立声明，再由上层按信用关系组合判断。

`ActionObject.base_on` 在这里很关键，因为它允许把下载纳入动作链，例如：

- `purchased -> download`
- `shared -> download`
- `download -> installed`

因此，下载证明不是一个孤立事件，而是传播路径里的一个中间节点。

### 安装证明

安装证明的核心语义不是“某个二进制被下载过”，而是：

> 某个主体确实把某个内容作为可执行或可使用对象接纳到了自己的环境里。

这在传播体系里明显高于“浏览”和“下载”，因为安装意味着：

- 更强的注意力投入。
- 更高的风险承担。
- 更明确的认可。

安装证明的设计意图主要有三层：

1. 它是内容传播的高价值反馈信号，可用于衡量“内容真的被使用了”，而不只是被看见了。
2. 它可以成为后续动作的基础。例如一个 `shared` 行为可以通过 `base_on` 指向某个 `installed` 行为，表示“我分享它，是因为我已经装过并用过它”。
3. 它可以参与信用计算。高信用主体签发的安装证明，比匿名或低信用主体签发的安装证明更有参考价值。

因此，安装证明不是许可证，也不是 DRM。  
它表达的是一个可验证传播事实：`某主体对象对某内容对象执行了 installed 动作`。

### 收录证明

收录证明是今天互联网上语义最明确、最成熟的一类传播证明。

它的核心语义是：

> 某个 curator 明确地把某内容收进了自己的目录、集合、榜单或推荐空间。

这不是“看过”，也不是“转发过”，而是一种更高成本的策展动作。

当前实现里，`InclusionProof` 直接表达了这套语义，例如会包含：

- `content_id`：被收录内容的 `ObjectId`
- `content_obj`：收录时引用的内容快照或关键字段
- `curator`：收录者 DID
- `collection`：收录进哪个集合
- `rank`：收录者对内容给出的排序或评级信号
- `editor`：如果 curator 是组织，哪些 editor 参与了策展
- `review_url`：外部评测或说明页
- `iat/exp`：收录声明的时间范围
- `meta`：例如真实性、稳定性、适用人群等附加维度

协议层里，收录证明的设计意图主要有四点：

1. 让“内容发现”从平台私有索引，变成可外带、可引用、可传播的策展声明。
2. 允许同一内容被多个 curator 独立收录，自然形成多链路视角。
3. 把“谁收录了什么”变成协议事实，便于 Agent 做去中心化聚合和权重计算。
4. 强调“收录者信用”比“收录数量”更重要。

因此，收录证明本质上是在为去中心化内容发现提供“目录层的注意力证明”。

### 分享证明

分享证明的核心语义不是“对象被复制了一份”，而是：

> 某个主体主动决定把某内容继续分发给新的对象、联系人、群组或传播空间。

在 AI-Native 网络里，分享是一次明确的注意力再分配。  
它回答的是“我决定把它传播给谁”，而不只是“我看过它”。

在现有设计里，分享证明通常不是单个对象，而是一组对象共同完成：

1. `cyact`
   - `action = "shared"`
   - 表达“分享动作已经发生”
   - `base_on` 可以指向更早的 `download`、`installed`、`purchased`、`viewed`

2. `cymsg`
   - 表达“分享是通过哪条消息、向哪些接收者、带着哪些上下文发生的”
   - `content.refs` 可以挂接被分享的内容对象

3. `cymsgr`
   - 表达“某条分享消息已被谁接收、接受、隔离或拒绝”
   - 它不是分享动作本身，但能为“分享已触达”提供侧证

4. `cyrel`
   - 当分享不是原样转发，而是摘录、节选、再创作、聚合时，`RelationObject` 可以表达新旧内容之间的关系

因此，分享证明最好被理解为一个小型证明包，而不是单条日志：

- `shared` 说明“有人做了传播动作”
- `MsgObject` 说明“传播面向谁、以什么形式传播”
- `MsgReceiptObj` 说明“传播是否触达”
- `RelationObject` 说明“传播的是原文、片段还是再创作版本”

协议层承认至少存在三种不同价值的传播：

1. 原样传播：重点是触达范围。
2. 带评论传播：重点是传播者的信用背书。
3. 再创作传播：重点是新旧内容关系与新的注意力增量。

### 内容消费证明

`内容消费证明` 是比“传播证明”更深一层的概念。  
它关注的不是“内容有没有被拿到”，而是：

> 用户是否真的在这个内容上投入了时间、注意力，甚至形成了后续行为。

从协议分层看，可以把它粗略理解为：

- `download`：拿到了内容。
- `installed`：把内容接纳进自己的环境。
- `shared`：愿意继续传播。
- `content consumption proof`：真实使用、真实停留、真实受益。

这一层目前仍是协议占位，但设计方向已经明确：

1. 它应建立在前述行为链之上，而不是凭空独立出现。
2. 它通常需要更强的隐私保护，因此很多场景更适合结合 `ZK` 或其它隐私机制。
3. 它需要更强的反机器人能力，不能只靠单边自报，而应依赖多角色、多阶段交叉验证。

所以，安装、下载、收录、分享这些证明，可以看作是 `内容消费证明` 之前的基础层。  
它们先把传播网络中的“谁发现、谁拿到、谁认可、谁继续扩散”表达清楚，之后才能进一步谈“谁真的用了它，以及这份使用如何参与经济分配”。

## 小结

`Content Network` 想建立的，并不是又一个中心化内容平台，而是一套公开的、可验证的内容传播语义：

- 用名字解决发现入口。
- 用 `NamedObject` 解决不可变内容定位。
- 用 `Curator`、`Sharer`、`Consumer` 把传播过程拆成可计算角色。
- 用传播证明和消费证明，把收益分配建立在真实行为之上。
- 用引用、二创和可计算内容，把“内容之间的关系”也变成网络的一等公民。

当这些能力具备后，一个内容网络才不只是“可下载”，而是真正“可发布、可传播、可评价、可结算、可再创作”。
