CYFS 是一个内容网络协议，核心定位是语义层的内容发布、内容的逻辑传播，以及围绕这套语义的标准对象构建和内容确定性约定。它在工程上被设计成对现有 HTTP 协议的最小必要扩展——留了很大的自由度给其他层次，我们自己在其他层次上各有各的实现，但协议本身不强加。
从协议角度看，CYFS 更多是"语义"层面的东西。它帮助我们在整个互联网的尺度上描述：内容网络长什么样、内容在节点之间如何流转。这里的"节点"指的是 Zone，不是具体设备——一个人的手机和电脑在 CYFS 看来通常是同一个节点。在这些节点之间流转的过程中会产生一些需要被协议定义的行为，其中很多与内容发布和经济循环有关：如何通过激励产生带用户签名的"事实存根"，让后续上层可以在这些存根之上构造不同的经济模型。协议本身只提供一个最小集合。我们内置的标准对象为什么是这些——答案是它们都是为了构造一个基于内容寻址、基于不可变内容的内容网络而设计的。我们始终在做减法。
身份这一层，我们尽可能遵循 W3C DID 标准，所有身份标识都写成 did:... 的标准格式。我们也有 DID 到域名的反向映射方法，基本就是把 DID 转成 ZoneId——里面有些细节属于传输实现层。我们的 BNS DID 扩展包含了一整套去中心化身份和认证体系，但通过 W3C DID 兼容，这部分内容不需要进入 CYFS 原始协议。我们也有基于 BNS DID 的原生转账机制，以及通过 DID Document 定义不同钱包地址的能力——我们理想的链是 USDB，但用户可以按 DID Document 的扩展机制选择自己的支付链路，包括与 HTTP 402 的兼容。这些都没有出现在 CYFS 协议文档里，因为它们属于更大的 HTTP 协议框架下的扩展，而不是 CYFS 本身的一部分。CYFS 本身只约定：用户访问受保护内容时需要提供购买证明，验证通过后发布方才传内容。协议不规定证明如何获得。
对于传统 P2P 协议里常讨论的问题，我们把它下沉到 cyfs-gateway 的 tunnel 框架。这一层有大量基础协议积累。我们的一个核心观点是：需要从面向 IP 地址的协议转向面向身份的协议。基于 DID 体系，我们有一个 tunnel 框架，类似 open_tunnel(target_did) 就可以打开一条到目标 Zone 的逻辑链路。这条链路的很多细节——跑在 TCP 连接上还是 TLS 上，还是 rTCP、rHTTP 上——全都属于更下层。CYFS 协议只假定一条已建立的 stream 通道，然后定义如何在这个通道上请求 Chunk、HTTP 头怎么写、返回的 Chunk 怎么与 ChunkId 验证——只定义这些。这里有个重要的设计意图：把可信传输这件事和中心化 CA 解绑，让 HTTPS 证书的等价物能通过 DID 获得。我们有自己基于区块链 DNS 的方案，但如果未来 W3C 有更高支持度的方案，我们也会拥抱——我们只是想在 tunnel 层保障：即使你不是一家公司、没有自己的证书，也一样可以在内容网络里可信地发布内容。
另一个常被问的问题是为什么没有 DHT、没有 BitSwap 式的多源交换协议。这里我们有一些非共识的判断。我们的经验横跨 P2P 和区块链两代网络。在我们看来 DHT 的历史使命已经完成——DHT 的核心目标就是给区块链做 bootstrap；只要一条链能稳定跑起来，就没有必要再用 DHT 这种不可靠、高延迟、非稳定性的系统来承担关键性配置读取。我们不会把什么东西真正需要的都放到 DHT 上，而是放到智能合约里。过去用 DHT 解决的真正去中心化问题，我们现在全部用智能合约解决——这是 CYFS 里完全没有 DHT 内容的原因。
在多源下载这块，我们的 FileObject 通过 sha256 切出 ChunkList，其实已经隐含了多源下载的核心底层支持。我们放弃了之前做过的各种喷泉编码和新传输协议——我们过去都定义过这些。但今天我们认为，作为最基础协议的 CYFS 不需要定义这些。它只需要定义：一个文件切成多个 Chunk。用户可以根据自己的实际情况用不同协议、从不同源拉这些 Chunk。这也跟今天的网速有关——以现在的带宽水平，传统 P2P 协议的 UDP 包反而是性能倒退。所以我们不定义 piece 粒度的 P2P 交换协议，只强调可验证的 Chunk 粒度。我们体系里其他层的协议会解决"如何高效多源获取一个 Chunk"的问题，CYFS 要解决的是"如何长期、公共地把一个大文件切成多个 Chunk，并且可以用不同协议把子 Chunk 拉回来"。
关于传统 Tracker 协议——我们现在愿意在 CYFS 里定义"逻辑广播 / 逻辑传播"，而不是协议层广播。我们从真实用户场景出发：CYFS 中的所有广播场景都对应真实的用户行为。比如一个收录者收录了一个信息，用户下载时至少能拿到"原作者 + 收录者"两个源。收录者为了评估热度，要求用户提交带签名的下载证明、安装证明等消费证明，并给予经济激励。这些签名过的消费证明本身，就是比传统 Tracker 更有价值的源信息来源——我们并没有传统意义上"谁拥有什么数据"的 Tracker 协议，但通过基于动作链的消费证明（每个 action 有 baseOn，形成 action 链），我们可以更灵活地建立可溯源的传播模型。这个设计意图也回答了为什么没有传统 Tracker——我们更希望站在真实用户场景的角度推导内容网络的传输过程，而不是站在协议抽象的角度推导用户行为。
